# QingNest 订阅定价与毛利模型

> 版本：2026-07-15。币种为人民币；美元按 `1 USD = ¥7.20` 估算。以下价格不含税，支付通道按收入的 3.5% 预留。Cloudflare、Supabase、Resend 的实际账单及汇率变化应每季度复核。

## 1. 经营结论

建议采用“免费获客 + 三档付费 + 按站点附加服务”，不要把请求次数作为首页核心卖点。

- 对外销售：全球 CDN、基础 DDoS 缓解、智能流量保护、热门站点加速、自定义域名、访问统计、版本回滚。
- 对内计量：HTTP 请求、R2 读取、存储、部署次数、Pages 加速槽位、域名数量。
- 热门站点达到阈值后自动迁移到独立 Pages，访客不再经过平台 Worker/KV/R2。
- 共享分发层先用 Cache API 吃掉 R2 和 KV：同一边缘节点首次访问最多回源一次，随后命中期内 R2/KV 均为 0；带内容哈希的资源按一年缓存，普通资源按一小时缓存，HTML 短缓存。
- 不因正常走红直接关站。先自动加速，再通知升级；只有攻击、违法、欠费或明确违反公平使用政策时才暂停。
- 域名绑定及换绑采用 24 小时冷却期。它主要减少证书、DNS/API 操作和滥用，并不能降低日常访问产生的 KV 读取。

推荐公开价格：`¥0 / ¥15 / ¥49 / ¥129 每月`。年付按 10 个月收费，即 `¥150 / ¥490 / ¥1,290`，不要做永久买断。

一次普通 R2 静态资源请求，如果同时执行 Worker、KV 读取和 R2 读取，超额后的理论成本约为：

`¥2.16 + ¥3.60 + ¥2.59 + CPU ≈ ¥8.5-10 / 百万 HTTP 请求`

若一次页面浏览平均产生 10 个 HTTP 请求，则 100 万 PV 可能产生约 1,000 万 HTTP 请求，底层超额成本约 `¥85-100`。因此必须缓存域名映射、采样统计，并把持续热门站点迁往 Pages。

### 2.3 Pages 与域名成本

- Pages 静态访问免费且不限请求量；不要给用户站点启用 Pages Functions。
- Free 每月 500 次部署，Pro 5,000 次，Business 20,000 次。
- Pages 项目默认软上限 100 个，需要提前申请提额。
- Free 单项目最多 100 个自定义域名；Pages 不支持通配自定义域名。
- Cloudflare for SaaS 包含 100 个 Custom Hostnames，之后约 `$0.10/域名/月`，即约 `¥0.72/域名/月`。
- 平台普通子域名继续使用一条通配 DNS，不应为每个冷站点创建 DNS 记录。

Pages 项目是稀缺资源，应包装成“热门站点加速槽位”，而不是所有免费站点默认占用一个项目。

- 免费版：让用户完成第一次发布，保留品牌，不承诺持续高流量容量。

现有数据库中的专业版 `100 项目 / 每日 500 次部署` 风险过高，建议调整为上表的 `20 项目 / 每日 100 次`。真正需要更多容量的客户走商务版或单独报价。

## 4. 流量与 DDoS 的销售方式

可以把请求限制改写成用户更容易理解的权益，但不能夸大为平台自建或绝对防御。

推荐对外文案：

- 全球 CDN 加速
- 基础 DDoS 缓解
- 智能流量保护
- 热门站点自动加速
- 恶意流量识别与限速
- HTTPS 自动证书

不推荐文案：

- “无限抗 DDoS”
- “绝不宕机”
- “无限流量”但同时隐藏公平使用限制
- 将正常访客限速包装成安全能力

所有套餐都应有基础 CDN 和 DDoS 缓解，因为这主要来自 Cloudflare。付费差异应放在自动加速优先级、加速槽位、统计、域名和支持上，而不是声称付费用户才受 DDoS 保护。

## 6. 超额处理规则

建议采用四级处理，不要一超额就关站：

1. 达到套餐内部公平使用量的 70%：控制台提醒，不影响访问。
2. 达到 100%：停止新发布或降低发布频率，网站继续访问。
3. 流量持续升高：自动迁移 Pages；付费用户按套餐加速槽位保留，免费用户作为临时保护处理。
4. 明确攻击、违法内容、欠费或迁移失败且威胁平台稳定：显示临时保护页并暂停站点，保留申诉入口。

正常爆火不应直接关站。它既是升级机会，也是用户口碑事件。应发送“网站流量增长，平台已自动加速”的通知，再推荐升级以保留加速和完整统计。

免费用户的临时 Pages 保护固定为 72 小时；热度持续时要求升级，否则回到共享分发层并按请求公平使用量限速，不删除用户数据。付费用户按套餐保留 7 天观察期，专业/商务套餐的加速槽位可持续保留。

## 7. 域名绑定与换绑政策

- 首次绑定不设冷却期，但必须完成所有权验证。
- 解绑后 24 小时内不得绑定到另一个项目。
- 同一域名每 24 小时最多换绑一次。
- 删除账号、误操作和安全事件由管理员人工解除。
- 换绑使用幂等任务，并保留旧映射到新证书生效，避免中断。

24 小时冷却期值得实施，但节省的主要是 KV 写入、Cloudflare API 调用、证书签发和客服成本。要减少 KV 读取，应先查 Worker Cache API，再用实例内短缓存保存 hostname 映射，并让 KV 只在两级缓存都未命中时读取。

## 8. 额度压缩与链路预算

优化目标不是让单项免费额度“刚好用完”，而是把每次用户动作的放大系数压到最低。所有新功能上线前都应填写下面的预算；超过预算必须说明原因并增加缓存、批处理或幂等控制。

| 链路 | Worker | KV | R2 | Supabase | 其他 |
|---|---:|---:|---:|---:|---:|
| 公开资源，边缘热命中 | 1 请求 | 0 | 0 | 0 | 0 |
| 公开资源，边缘冷命中 | 1 请求 | 最多 1，实例热时 0 | 最多 1 个成功对象；路径探测失败时可能更多 | 仅 KV 丢失时兜底，正常为 0 | 0 |
| 同一页面后续静态资源 | 每资源 1 请求 | 命中期内 0 | 命中期内 0 | 0 | 0 |
| 私有预览，同一 Worker 实例 | 每资源 1 请求 | 首个资源最多 1，随后 0 | 每资源最多 1 个成功对象 | 0 | 0 |
| 控制台普通页面切换 | 每个未缓存 API 1 请求 | 0 | 0 | 只请求缺失数据；60 秒内重复读取为 0 | 0 |
| 发布 | 按 API 步骤 | 映射最终写 1 次 | 每个文件写 1 次 | 元数据批量写，禁止逐文件往返 | 审核/邮件只在状态变化时触发 1 次 |

具体执行规则：

- 先查 Cache API，再查域名映射；禁止为了路由每个静态资源都读取 KV。
- Cache API 的边缘 TTL 继承资源语义：哈希资源一年、普通资源一小时、HTML 60 秒、404/下架/审核状态 15 秒。
- 缓存键忽略查询字符串，因此用户静态站点不能依赖 query 参数返回不同文件内容；查询参数只用于前端逻辑和缓存刷新标记。
- 同一部署的文件元数据一次批量写入 Supabase；访客请求永远不查 Supabase，不保存逐次访问日志。
- 前端持有当前 access token，API 调用不再逐次执行 `getSession()`；token 刷新由 Auth 状态事件更新。
- 发布、换绑、邮件、迁移必须使用幂等键或冷却锁，重试不能重复消耗 R2 Class A、KV 写入、证书或邮件额度。
- 404 必须负缓存。恶意扫描路径时，单一 URL 在 15 秒内只允许产生一次回源探测。
- 热门站点迁至 Pages 后，公开访问预算为 Worker 0、KV 0、R2 0、Supabase 0；仅部署和聚合统计消耗额度。

这里的“R2 0 次”指命中同一 Cloudflare 边缘缓存或迁移 Pages 后的请求。首次访问某个边缘节点、缓存过期、重新发布或预览仍可能读取 R2，运营报表必须区分冷回源和热命中，不能把局部命中宣传成全局永久 0 次。

## 9. 热门站点自动晋升

当前实现同时监控共享 Worker 与 Pages 主机名流量。连续两个 15 分钟窗口达到 15,000 请求才晋升，避免偶发爬虫制造 Pages 项目；降温需要连续 96 个窗口（24 小时）低于 1,500 请求，且临时保护期已经结束。

晋升流程为 `R2 共享分发 -> 创建 Pages -> 上传当前部署 -> 验证域名/证书 -> 切换路由 -> 保留 R2 回滚版本`。至少观察 24 小时，不自动频繁降级。

Analytics Engine 统计仍在 Worker 上的站点；迁移 Pages 后由 Cloudflare GraphQL 按主机名读取区域请求聚合。Supabase 只保存站点状态和窗口汇总，不保存逐次访问日志。

自动晋升只接受 Pages 兼容资产：单文件不超过 25 MiB、文件不超过 20,000 个、总直传包不超过 90 MiB。超过条件时保持共享分发并记录失败原因，不会在爆火后切换到一个无法承载该部署的 Pages 项目。

## 10. 单客毛利模型

以下是经营预算，不是 Cloudflare 的逐用户账单。把支付费、正常资源消耗、热门站点风险准备金和基础支持一起计入。

| 套餐 | 收入 | 支付费 3.5% | 正常资源预算 | 加速/风险准备金 | 支持准备金 | 贡献毛利 | 贡献毛利率 |
|---|---:|---:|---:|---:|---:|---:|---:|
| 入门版 | `¥15.00` | `¥0.53` | `¥1.50` | `¥0.80` | `¥1.50` | `¥10.67` | `71.1%` |
| 专业版 | `¥49.00` | `¥1.72` | `¥5.00` | `¥5.00` | `¥4.90` | `¥32.38` | `66.1%` |
| 商务版 | `¥129.00` | `¥4.52` | `¥15.00` | `¥14.00` | `¥12.90` | `¥82.58` | `64.0%` |

固定基础设施按 `¥366/月` 计算时：

- 约 27 个入门版用户可覆盖固定成本；
- 或约 10 个专业版用户可覆盖固定成本；
- 或约 4 个商务版用户可覆盖固定成本。

更现实的组合示例：100 个付费用户中，60 个入门版、35 个专业版、5 个商务版。

| 项目 | 金额/月 |
|---|---:|
| 套餐收入 | `60×19 + 35×59 + 5×169 = ¥4,050` |
| 支付费预留 | `-¥142` |
| 单客资源/风险/支持预算 | `-¥1,105` |
| 固定基础设施 | `-¥366` |
| 预计经营贡献 | `约 ¥2,437/月` |
| 预计经营贡献率 | `约 60.2%` |

该贡献尚未扣除税费、开发者工资、市场投放、退款、坏账和公司管理成本。成熟后应把整体基础设施成本控制在收入的 10%-15%，客服与退款控制在 10%以内，支付费约 3%-4%，最终保持 60%左右的产品毛利。

## 11. 成本失控红线

后台必须设置以下告警：

- Workers 当月请求达到包含额度的 70%、90%。
- KV 读取达到 70%、90%，KV 写入异常增长。
- R2 Class B 达到 700 万、900 万次。
- Pages 项目达到 70、85、95 个。
- Pages 月部署达到额度的 60%、80%、90%。
- Supabase 数据库达到 6GB，出口达到 180GB。
- Resend 达到套餐邮件量的 80%。
- 单个免费用户占平台当日成本超过 `¥1` 或单个付费用户连续两月成本超过其收入的 25%。

若某付费客户连续两个月直接成本超过其套餐收入的 25%，应推荐升级、购买加速槽位或转为定制报价，而不是继续用统一套餐补贴。

## 12. 实施顺序

1. 先上线四档套餐、24 小时域名换绑冷却和发布次数限制。
2. 固化链路额度预算并接入命中率、冷回源和外部调用计数。
3. 接入 Analytics Engine，只保存采样后的聚合统计，不写逐次访问数据。
4. 验证边缘缓存：热命中 R2/KV 为 0，哈希资源一年、普通资源一小时、HTML 60 秒。
5. 用测试域名跑通单站点 Pages 晋升、证书、路由和回滚。
6. 上线专业版 1 个、商务版 3 个热门站点加速槽位。
7. 接近 70 个 Pages 项目时提交提额申请。

## 13. 分阶段 Pages 容量预算与管理员预警

- `$5 起步期`：Workers Paid + Pages Free。Pages 每月 500 次部署，建议提醒/严重阈值为 `60% / 80%`，为首次迁移、失败重试和免费站临时保护保留更大余量。
- `$5 稳定期`：仍使用 Pages Free，业务链路稳定后改为 `70% / 90%`；月部署超过 350 次或第 20 天前超过 300 次时评估升级。
- `$5 + $25 扩容期`：Pages 预算按 5,000 次/月管理，默认 `70% / 90%`；升级前必须以 Cloudflare 控制台实际显示的账户额度为准。
- 免费版只提供最长 72 小时临时保护；入门版不保证长期槽位；专业版保证 1 个槽位；商务版保证 3 个槽位。平台发布次数与 Pages 同步次数必须分别计量。
- 管理后台“容量”页显示已用次数、额度和即时比例。预设和每个数值均可调整；键入只在浏览器重算，保存时一次写入。
- 定时预警只写管理员审计事件，使用冷却窗口去重；绝不写用户通知表，也不在用户侧弹窗。未接入 Cloudflare Analytics 的账单指标必须标注为待采样，不得用平台估算冒充账单数据。
